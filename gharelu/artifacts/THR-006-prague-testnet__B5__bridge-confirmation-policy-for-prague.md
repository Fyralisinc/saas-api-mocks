# RFC: Bridge Confirmation Policy for Prague

## Summary

This RFC defines how `strata-bridge` should represent and enforce Bitcoin confirmation depth for the Prague testnet, and how that policy should be surfaced across `strata`, `strata-bridge`, `strata-p2p`, `checkpoint-explorer`, `alpen-dashboards`, and `alpen-faucet`.

The immediate change is to stop treating Prague confirmation depth as an implicit bridge-local default. Instead, Prague bridge config should carry an explicit confirmation policy object with separate values for deposit recognition, withdrawal finalization, and explorer/dashboard display. The bridge runtime should consume a single canonical config artifact generated by infra, while other repos should either import the generated artifact or consume it through deployment metadata.

For Prague, I propose:

```toml
[bridge.confirmations]
network = "prague"
deposit_min_depth = 6
withdrawal_finality_depth = 12
display_safe_depth = 6
policy_version = 1
```

The important part is not the exact numbers alone. The policy must be versioned, explicit, and shared. During the B5 staging smoke tests, person:ProofOfKeags and I found that deposit and withdrawal paths were reading semantically similar confirmation settings from different repos, with different assumptions about defaults. That made it too easy for the bridge to accept a deposit before the UI, explorer, and dashboards agreed it was safe.

person:prajwolrg remains driver for Prague support. This RFC is an amendment from the bridge testing pass and should be treated as a blocking Prague readiness item for public bridge use.

## Motivation

Prague has reached the point where the hard problems are no longer just protocol correctness in isolation. We now need the bridge, dashboards, faucet, explorer, and public docs to describe the same chain state at the same time.

The confirmation-depth question exposed a coordination problem:

- `strata-bridge` needed a depth threshold for accepting Bitcoin deposits.
- Withdrawal smoke tests needed a stronger threshold before treating a withdrawal as externally final.
- `checkpoint-explorer` and `alpen-dashboards` needed to display confirmation state without accidentally implying finality earlier than the bridge would enforce it.
- `alpen-faucet` and public docs needed a user-facing statement like “wait N confirmations” that matched what the bridge actually does.
- Infra wanted one rollout knob, not a per-repo hunt through config files.

During staging tests in B5, person:ProofOfKeags and I ran deposits and withdrawals against infra provided by person:krsnapaudel. The bridge path itself behaved mostly as expected, but the config shape did not. Confirmation depth existed as duplicated constants and defaults. Some code paths treated the value as “minimum Bitcoin depth to observe a deposit.” Others treated it as “final enough for bridge UX.” Those are not the same policy.

This matters even on testnet. Prague is the first environment where external users, devrel, and internal protocol testing all meet. If person:john-light or person:bewakes is debugging a public bridge issue, they should not need to know which repo encoded which interpretation of “confirmations.” If person:MdTeach or person:prajwolrg changes a Prague network parameter, bridge behavior should not silently diverge from explorer behavior.

The goal is to make confirmation policy boring: explicit in config, reviewed once, rolled out once, and observable everywhere.

## Detailed design

### Policy model

We should represent Prague bridge confirmation policy as a structured object, not a single integer.

Required fields:

```toml
[bridge.confirmations]
network = "prague"
deposit_min_depth = 6
withdrawal_finality_depth = 12
display_safe_depth = 6
policy_version = 1
```

Field meanings:

`network` identifies the target network profile. For this RFC, the only required value is `prague`.

`deposit_min_depth` is the minimum Bitcoin confirmation depth before `strata-bridge` recognizes a deposit as eligible for bridge processing. This is a bridge consensus-facing operational threshold, not just a UI hint.

`withdrawal_finality_depth` is the minimum Bitcoin confirmation depth before bridge tooling reports a withdrawal path as externally final for smoke tests, dashboards, and operator runbooks. This is intentionally allowed to be higher than `deposit_min_depth`.

`display_safe_depth` is the threshold used by explorer, dashboards, faucet status, and docs when they need a simple “safe after N confirmations” value. For Prague this should initially match `deposit_min_depth`, but it is kept separate so we do not overload protocol behavior with UX copy.

`policy_version` is a monotonically increasing integer. Any change to the policy values must bump this number. The version should be logged by bridge startup and exported in dashboard metadata.

### Prague defaults

For Prague, this RFC sets:

- `deposit_min_depth = 6`
- `withdrawal_finality_depth = 12`
- `display_safe_depth = 6`

The reason for 6 deposits is practical: it is conservative enough for a public Bitcoin testnet-like environment while keeping smoke tests and faucet-funded flows usable. A lower value, especially 1 or 2 confirmations, makes demo flows faster but creates bad habits in bridge code and public docs. A higher value, like 12, slows down every deposit test and makes staging iteration worse.

The reason for 12 withdrawal finality is that withdrawals are the harder path to explain and recover from. A deposit accepted too early can usually be reclassified or delayed internally. A withdrawal shown as final too early creates worse operator and user confusion. Prague should bias toward clarity here.

These values can be revisited after Prague has a week of bridge traffic, but the mechanism should not change.

### Canonical config artifact

Infra should produce a canonical Prague network config artifact as part of deployment. The bridge should not maintain a second hand-written copy.

The artifact can be TOML or JSON; I do not care about the serialization as long as it is typed, versioned, and generated from the same source that person:krsnapaudel uses for rollout. The required property is that `strata-bridge` reads confirmation policy from the canonical artifact in Prague deployments.

Proposed ownership:

- person:krsnapaudel owns generation and rollout wiring.
- person:Rajil1213 owns bridge config parsing and runtime enforcement.
- person:ProofOfKeags owns smoke test coverage for deposit and withdrawal behavior.
- person:prajwolrg reviews the protocol-facing semantics.
- person:MdTeach reviews compatibility with Prague node and sequencer assumptions.

### Bridge runtime behavior

On startup, `strata-bridge` must:

1. Load the confirmation policy object.
2. Reject Prague config if any required field is missing.
3. Reject config where `deposit_min_depth < 1`.
4. Reject config where `withdrawal_finality_depth < deposit_min_depth`.
5. Log `network`, all depth values, and `policy_version`.
6. Expose the active policy through bridge health or metrics.

Deposit handling must use `deposit_min_depth` and no other local default.

Withdrawal status reporting and smoke-test finality checks must use `withdrawal_finality_depth`.

If the bridge is running against Prague and no explicit confirmation policy exists, startup should fail. Silent fallback is the bug class this RFC is trying to remove.

### Cross-repo consumers

`checkpoint-explorer` should display confirmation progress against `display_safe_depth` for deposits and against `withdrawal_finality_depth` for withdrawals when the withdrawal context is available. It should also display the active policy version in internal/debug views.

`alpen-dashboards` should track the active policy version as a label or annotation. Dashboard alerts should include enough detail to answer: “is this service using the current Prague policy?”

`alpen-faucet` does not need to enforce bridge policy, but any bridge-related faucet message should use `display_safe_depth`.

`strata-p2p` should not carry bridge confirmation constants. If it needs network metadata for diagnostics, it should consume the same generated artifact or receive the value from deployment metadata.

`strata` should treat this as environment configuration, not protocol logic, unless person:prajwolrg identifies a consensus-facing dependency that I am missing.

### Test plan

Bridge smoke tests should include:

- Deposit with depth below `deposit_min_depth` remains pending.
- Deposit at exactly `deposit_min_depth` becomes eligible.
- Withdrawal below `withdrawal_finality_depth` is not reported final.
- Withdrawal at exactly `withdrawal_finality_depth` is reported final.
- Prague startup without confirmation policy fails.
- Prague startup with `withdrawal_finality_depth < deposit_min_depth` fails.
- Metrics or health endpoint includes `policy_version`.

## Drawbacks

This adds ceremony to something that used to be a simple integer. For local development, engineers will need a valid confirmation policy even when they do not care about bridge timing. We can reduce friction with a local profile, but we should not make Prague permissive.

The split between deposit, withdrawal, and display depths may feel over-modeled for testnet. I think the distinction is worth it because the existing ambiguity already caused review churn. A single `confirmations = 6` value is compact, but it does not say what it means.

Failing bridge startup on missing policy can slow down infra rollout if deployment metadata lags code. That is intentional for Prague public readiness, but it means person:krsnapaudel and bridge owners need to coordinate the merge order.

## Alternatives considered

### Keep one integer

The simplest option is:

```toml
confirmations = 6
```

I do not recommend this. It preserves the ambiguity between deposit eligibility, withdrawal finality, and UI display. The B5 issue was not that we picked the wrong integer; it was that different repos interpreted the integer differently.

### Hard-code Prague bridge defaults

We could hard-code Prague defaults in `strata-bridge` and let other repos copy them. This is the current failure mode with a cleaner constant name. It does not give infra one rollout knob and does not prevent drift.

### Use only deployment environment variables

Environment variables are easy for infra, but weak for typed policy. A set like `PRAGUE_DEPOSIT_CONFIRMATIONS=6` and `PRAGUE_WITHDRAWAL_CONFIRMATIONS=12` is workable, but harder to version and easier to partially configure. If we use env vars, they should populate the same typed policy object and still require `policy_version`.

### Make all depths 12

This is conservative and simple, but too slow for deposit testing and faucet-funded public flows. Prague needs realistic bridge behavior without making every deposit smoke test wait longer than necessary.

### Make all depths 1 or 2

This improves demo speed but weakens the signal of Prague testing. It would also force docs and dashboards into a policy we would never want users to internalize. I do not think Prague should optimize for the fastest happy-path demo at the expense of bridge semantics.

## Open questions

Should `display_safe_depth` exist as a first-class field, or should public-facing consumers derive it from `deposit_min_depth` until we have evidence that they need to differ?

Where exactly should the canonical artifact live? My preference is infra-owned generation with checked-in schema validation, but person:krsnapaudel should decide the rollout path that minimizes deployment drift.

Do we want policy changes to require protocol review from person:prajwolrg every time, or only when the value crosses a defined bound? For Prague, I think any change before public launch should get explicit review.

Should `checkpoint-explorer` show both deposit and withdrawal thresholds, or only the threshold relevant to the object being viewed? I prefer contextual display to avoid making the UI read like internal config.

Do we need a backwards-compatible bridge config path for older staging deployments, or can Prague be the boundary where missing confirmation policy becomes a hard error? My recommendation is hard error for Prague and permissive warning only for non-Prague local profiles.
